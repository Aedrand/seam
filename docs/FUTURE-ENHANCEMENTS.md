# Future Enhancements

## Priority Implementation

Items we want to build next. These came up during MVP development and are high-value, low-complexity additions.

### Repo-to-Workspace Linking

**The problem:** When an agent starts a session, it has to manually set or join a workspace. If you work across multiple projects, you're calling `set_workspace` every time you switch repos.

**What it could look like:**
- A new tool `link_workspace(repo, workspace)` that maps a repo identifier to a workspace name
- The repo identifier could be the git remote URL (e.g., `github.com/Aedrand/seam`) or the directory basename
- On session start, the MCP client passes a repo identifier (via query param or tool call), and the server auto-sets the active workspace based on any existing link
- Links are stored per user — different users can link the same repo to different workspaces if needed
- `unlink_workspace(repo)` to remove a mapping
- Falls back to normal `set_workspace` if no link exists

**Why it matters:** Eliminates the most common manual step in using Seam. You open a project, the agent is already in the right workspace. Zero friction.

**Design questions to resolve:**
- What identifies a "repo"? Git remote URL is most reliable, but what if the agent isn't in a git repo?
- Should the link be set by the agent automatically (e.g., "I'm in repo X, link it to this workspace"), or only via explicit user instruction?
- Could the MCP client pass the repo identifier automatically via a query parameter (like `?workspace=` today), so the server resolves it without a tool call?

Ideas that are out of scope for now but worth revisiting as Seam matures.

## Invite-Based Registration

**The limitation today:** Anyone with the bootstrap token can register. For small trusted teams this is fine. For larger groups or servers on the public internet, there's no way to control who joins.

**What it could look like:**
- The first user registers with the bootstrap token
- Subsequent users need a single-use invite token generated by an existing user
- Invite tokens are workspace-scoped — redeeming one registers you (if needed) and adds you to the workspace in one step
- Handles both new users (register + join) and existing users (just join) transparently

**Server config:** A `REGISTRATION_MODE` flag — `open` (current behavior) or `invite` (invite tokens required).

**Why it's deferred:** Adds invite generation, redemption, and single-use tracking. More auth machinery than a lightweight MVP needs. Open mode with a shared token is sufficient for the initial audience of small trusted teams.

**When to revisit:** When the server is exposed on the public internet, or when teams larger than 5-10 people start using it.

## Multi-Workspace Agent Access

**The limitation today:** An agent operates in one workspace at a time. Switching projects means changing the MCP config.

**What it could look like:**
- Agent connects with access to multiple workspaces
- Tools accept an optional workspace parameter
- Agent can read context from a secondary workspace without switching

**Why it's deferred:** Adds "which workspace?" ambiguity to every operation. For monorepo or cross-project work, separate MCP server entries per workspace is workable.

**When to revisit:** When users report frequently needing context from multiple projects in a single session.

## Section Staleness

**The limitation today:** Sections carry timestamps but nothing flags or nudges cleanup of old content. A "current-status" section from two weeks ago is visible but potentially misleading.

**What it could look like:**
- Index marks sections as stale after a configurable period
- Stale sections display a warning when read
- Optional: agents are prompted to update or confirm stale sections they own

**Why it's deferred:** Timestamps are visible in the index. Agents can judge recency themselves. Adding staleness logic means defining thresholds and notification behavior — complexity for a problem that may not materialize at small scale.

**When to revisit:** When projects run long enough (weeks/months) that outdated sections actively mislead agents.

## Section Size Limits

**The limitation today:** No enforcement on section length. An agent could write a 10,000-token section that blows out context when read.

**What it could look like:**
- Soft limit: server warns when a section exceeds a threshold
- Hard limit: server rejects writes above a maximum
- Guidance in tool descriptions (current approach) may be sufficient

**Why it's deferred:** Tool description guidance ("each section should cover one topic and fit comfortably in context") is the lightweight approach. Hard limits add configuration and edge cases. See if agent judgment is sufficient first.

**When to revisit:** If agents consistently write oversized sections that cause context problems.

## CRDT-Based Conflict-Free Writes

**The limitation today:** When two agents write the same section concurrently, the second writer gets a version mismatch error and must re-read, mentally merge, and rewrite. This works but is sequential — one agent waits while the other finishes.

**What it could look like:**

Instead of whole-section version locking, use CRDTs (Conflict-free Replicated Data Types) to merge concurrent writes automatically. Different parts of a section get the merge strategy matching their access pattern:

- **Append-only content** (observations, status updates) — G-Sets, just union everything. Two agents adding different notes never conflict.
- **Single-value fields** (section description, author) — LWW-Registers, most recent timestamp wins.
- **Structured lists** (items that can be added and removed) — OR-Sets, concurrent add/remove handled safely.

The math guarantees that no matter what order writes arrive, the result is the same. No blocking, no "re-read and try again," no coordinator.

**Inspiration:** [Kli](https://github.com/kleisli-io/kli) — a Common Lisp MCP server that applies this approach to agent task graphs. Each field gets the CRDT type matching its concurrent access pattern. Event-sourced (append-only JSONL), state reconstructed by replay. See also [Yjs](https://github.com/yjs/yjs) and [Automerge](https://automerge.org/) for mature CRDT libraries.

**Why it's deferred:** Seam's version checking is simpler and sufficient for 2-5 agents. CRDT-based merging adds significant architectural complexity and is only worth it when version conflicts become frequent enough to slow teams down. At small scale, re-read-and-rewrite is a minor inconvenience, not a bottleneck.

**When to revisit:** When teams report frequent version conflicts on the same section, or when Seam scales beyond small teams to larger groups where sequential locking becomes a real bottleneck.

## CLAUDE.md Startup Instruction

**The limitation today:** The MVP relies on MCP tool descriptions to prompt agents to call `get_index()` on startup. This works well for clients that follow tool descriptions (Claude Code does), but may not be reliable across all MCP clients.

**What it could look like:**
- A one-liner in CLAUDE.md: `On startup, call get_index() from the Seam MCP server to load shared project context.`
- Could be added automatically by a `seam init` command or documented as a recommended setup step

**Why it's deferred:** Tool descriptions are the lighter-touch approach and align with the "zero config beyond installing the server" goal. Worth testing whether they're sufficient before adding a manual setup step.

**When to revisit:** If agents don't reliably call `get_index()` on startup from tool descriptions alone, particularly across non-Claude MCP clients.

## Graduated Staleness via Decay

**The limitation today:** Section staleness (above) proposes a binary flag — stale or not stale after N days.

**What a more nuanced version could look like:**

Instead of binary staleness, sections carry a decay model inspired by [SBP's digital pheromone system](https://github.com/AdviceNXT/sbp). Each section has a confidence score that degrades continuously over time:

- **Exponential decay** — confidence halves every N days (configurable per section type)
- **Reinforcement** — when an agent reads a section and confirms it's still accurate, the decay clock resets. Frequently validated context stays strong; abandoned context fades.
- **Compute-on-read** — never store current confidence, always derive it from last-confirmed timestamp + decay model. Eliminates stale-state bugs.
- **Evaporation threshold** — below a configurable floor, sections are flagged in the index as low-confidence rather than removed

This gives agents a continuous signal ("this section is 0.9 confidence" vs "0.3 confidence") rather than a binary "fresh/stale" flag, enabling more nuanced judgment about how much to trust older context.

**Why it's deferred:** Adds complexity to the data model and the index rendering. The simpler staleness flag (or just timestamps with agent judgment) is sufficient for MVP. Decay models are a sophisticated optimization for long-running projects.

**When to revisit:** When the simpler staleness mechanism proves too blunt — either flagging things too aggressively or not aggressively enough.

## LLM-Assisted Merge on Version Conflict *(higher priority)*

**The limitation today:** When a version check fails, the agent re-reads the current version and rewrites, manually incorporating the other agent's changes. This works but relies entirely on the agent's judgment about what to keep and what to merge.

**What it could look like:**
- On version conflict, the server returns both the agent's attempted write and the current version
- The agent (or a lightweight helper) performs a semantic merge — keeping the best of both, deduplicating, resolving contradictions
- Inspired by [claude-brain's](https://github.com/toroleapinc/claude-brain) two-stage merge: deterministic for structured data, LLM for unstructured prose

**Why it's deferred:** The current approach (re-read, rewrite) is simple and sufficient. The agent already has the intelligence to merge — it just needs to see the current version. Adding a formal merge step adds complexity to the tool flow without clear benefit at small scale.

**When to revisit:** When version conflicts become frequent enough that the manual merge feels like friction, or when the quality of agent-driven merges proves inconsistent.

## Meta-Tool Pattern

**The limitation today:** Seam exposes 4 tools (`get_index`, `read_section`, `write_section`, `delete_section`). This is manageable, but if Seam grows to include more tools (handoffs, messaging, consensus from the broader agent-collab vision), the tool list could get crowded in the agent's context.

**What it could look like:**
- Only 2-3 tools visible to the agent: `discover_tools`, `describe_tool`, `execute_tool`
- All other tools called indirectly through `execute_tool`
- Keeps the agent's tool list clean while exposing arbitrary functionality
- Inspired by [Forgetful's](https://github.com/ScottRBK/forgetful) meta-tool pattern (3 visible tools, 56 accessible underneath)

**Why it's deferred:** 4 tools is fine. This pattern only matters if Seam grows significantly in scope, which goes against the lightweight value.

**When to revisit:** If Seam's tool count exceeds ~8-10 and agents start struggling with tool selection.

## Cross-Tool Export *(higher priority)*

**The limitation today:** Seam serves context via MCP. Only MCP-connected agents can access it. Agents using tools without MCP support (or users who just want to read the shared context) have no access.

**What it could look like:**
- Seam exports its sections as static files in tool-native formats: CLAUDE.md fragments, .cursorrules, AGENTS.md, Copilot instructions
- A `seam export` CLI command generates these files from the current server state
- Could integrate with [rulesync](https://github.com/dyoshikawa/rulesync) (918 stars, 25+ tool support) as an export target
- Sections become available to any AI coding tool, not just MCP-compatible ones

**Why it's deferred:** MCP compatibility covers the primary audience (Claude Code, Cursor, Codex). Cross-tool export adds a build/sync step that complicates the "just install the MCP server" simplicity. Worth exploring only if adoption is limited by MCP-only access.

**When to revisit:** When users on non-MCP tools (or mixed-tool teams) want access to shared context, or when rulesync integration is requested.

## Section Version History

**The limitation today:** Only the current version of a section is stored. When a section is updated, the previous content is lost. There's no way to see what changed between versions or recover an earlier state.

**What it could look like:**
- Store previous versions of each section alongside the current one
- `read_section` could accept an optional version parameter to retrieve a specific historical version
- A `section_history(name)` tool could return a changelog of versions with diffs, authors, and timestamps
- Useful for debugging ("what did this say before?") and for the LLM-assisted merge enhancement

**Why it's deferred:** Current-state-only is simpler — one row per section, no history table, no storage growth concerns. At MVP scale, if something goes wrong with a section, an agent can just rewrite it.

**When to revisit:** When teams want to understand how shared context evolved, or when the LLM-assisted merge enhancement needs access to prior versions.

---

## Soft Deletes

**The limitation today:** Deleting a section permanently removes it from the database. There's no recovery if a section is deleted by mistake or if an agent incorrectly decides context is no longer needed.

**What it could look like:**
- Deleted sections are marked as deleted rather than removed
- A grace period (e.g., 7 days) before permanent removal
- A `restore_section(name)` tool to recover recently deleted sections
- Deleted sections hidden from the index but recoverable by name

**Why it's deferred:** Hard deletes are simpler and sufficient for small teams where trust is high. Sections are written by agents for agents — if something is deleted incorrectly, it can be rewritten. Adding soft delete means managing tombstones, cleanup schedules, and a recovery tool.

**When to revisit:** When accidental deletions become a real problem, or when teams want an undo safety net.

---

## Session Fingerprinting for Conflict Detection

**The limitation today:** Conflict detection is version-based — you only discover a conflict when you try to write. There's no proactive warning that another agent is likely to modify the same section.

**What it could look like:**
- Agents register behavioral fingerprints: what sections they've read, what they're working on, what files they're touching
- The server compares fingerprints and warns when two agents have high overlap (cosine similarity > 0.8)
- Warning on `read_section`: "agent-b is also actively working with this section"
- Inspired by [Kli's](https://github.com/kleisli-io/kli) session fingerprinting — behavioral signatures detect likely collisions without explicit locking

**Why it's deferred:** Version checking handles actual conflicts. Proactive warnings add a presence/tracking layer that increases server complexity. At 2-5 agents, the collision probability is low enough that reactive detection is sufficient.

**When to revisit:** When teams scale to the point where version conflicts are frequent and the reactive "try, fail, re-read" cycle becomes disruptive.
